    As we can see from the title of this page as well as the image above, SOLID is not referring to matter, but to an abbreviation that stands for design principles in object oriented programming. Each letter in S.O.L.I.D refers to a principle for development.

    1) S: Single Responsibility Principle

  • The single responsibility principle simply means the code should have one responsibility or function to perform and one change. We can see an example below:

    • def divide_numbers(value_a, value_b)

    The code above is a single responsibility function which defines division. This code can only be used to divide numbers and the values are the only things that change but do not modify the code to have another responsibility.

    2) O: Open-Closed Principle

  • The open-closed principle means classes, modules and functions should be open to extensions but closed to modifications. We can see an example below:

    • def convert_args_to_tuple_of_float(values):

    list_values_float = []

    for item in values:

    list_values_float.append(float(item))

    return tuple(list_values_float)

    In the above code, we see an example of an extension. We are not modifying the code, but we are giving it an extension in terms of items in values to be listed in float format.

    3) L: Liskov Substitution Principle

  • The LSP emphasizes the definition of polymorphism defined in the OOP article. Let's recap and look at an example of polymorphism:

    • Polymorphism: Using the same function for different types of objects.

    def get_result(self):

    return self.value_a + self.value_b

    In the polymorphic code above, all we need to realize to understand LSP is that the function "get_result" is being defined by the object "self"; thus, the function remained the same but with an interchangeable object "self" which, even if modified, would still perform the function of getting a result.

    4) I: Interface Segregation Principle

  • The ISP means no code should have to rely on methods it does not use. To prevent this, we should break down large interfaces into smaller ones. We can think of a combination of the single responsibility principle in which the code has one responsibility with the appropriate abstraction, the pulling of higher-module data into lower ones. An example would be tuples.

    • def __init__(self,values: tuple)

    self.values = Calculation.convert_args_to_tuple_of_float(values)

    Tuples can hold large amount of functions/data from another class (abstraction) with one responsibility (single responsibility principle)

    5) D: Dependecy Inversion Principle

  • The DIP means high-level modules should not depend upon low-level modules. We can think of the parent classes vs. child classes here. Thus, we will look at abstraction, also defined in the OOP article.

    • Abstraction: This is the abstract class or the parent class wherein a set of methods are created to be used in child classes. Example:

    """This is my calculation base class / Abstract Class"""

    class Calculation:

    def __init__(self,value_a, value_b):

    self.value_a = value_a

    self.value_b = value_b

    In the code above we see the parent class which defines the main method to be used throughout the calculator programming (child classes). We see here what the DIP is talking about, the child classes do not have to rely on any other code but the parent class.
